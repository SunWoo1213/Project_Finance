# AI Invest 발표자료용 상세 번호 표 구성안

문서 목적 : 이 문서는 `PROJECT_PAGE_SEQUENCE_DETAIL_SPEC_AND_SCRIPT.md`와 `PROJECT_FUNCTION_DETAIL_SPEC.md`를 다시 읽고, 발표 PPT 각 페이지에 넣을 표를 더 자세하게 재구성한 자료이다.

작성 기준 : 발표 순서 21개 페이지를 기준으로, 각 페이지 안에서 설명해야 할 요소를 세부 번호로 나누었다. 기존 표가 페이지의 전체 흐름을 요약했다면, 이 문서는 화면 요소, 사용자 행동, 프론트엔드 처리, 백엔드/API 처리, 발표 포인트를 단계별로 설명한다.

사용 방법 : 각 페이지 표에서 `세부 번호` 순서대로 발표하면 된다. PPT 공간이 부족하면 `발표 포인트` 열을 중심으로 줄이고, 발표자 노트에는 `기술 근거` 열을 넣으면 된다.

## 2. 페이지별 상세 번호 표

### 1페이지. 로그인 화면

| 세부 번호 | 설명 대상 | 화면/사용자 행동 | 내부 처리/API | 발표 포인트 |
|---|---|---|---|---|
| 1-1 | 화면 진입 | 사용자가 `/login` 화면으로 이동한다. | React Router가 `Login.jsx`를 렌더링한다. | 로그인은 AI 리포트와 커뮤니티 보호 기능을 사용하기 위한 첫 단계이다. |
| 1-2 | 입력 UI | 이메일 입력 필드와 비밀번호 입력 필드에 값을 입력한다. | 공통 `InputField` 컴포넌트를 사용한다. | 사용자 인증에 필요한 최소 정보만 입력받는다. |
| 1-3 | 프론트 검증 | 로그인 버튼을 누르면 입력값을 먼저 검증한다. | `react-hook-form`과 Zod로 이메일 형식, 비밀번호 입력 여부를 검사한다. | 잘못된 요청을 서버로 보내기 전에 프론트에서 1차로 차단한다. |
| 1-4 | 로그인 요청 | 검증 성공 후 로그인 API를 호출한다. | `POST /api/auth/login`, `application/x-www-form-urlencoded`, `username`, `password` 전송 | OAuth2PasswordRequestForm과 호환되는 방식으로 로그인한다. |
| 1-5 | 백엔드 인증 | 서버가 이메일로 사용자를 찾고 비밀번호를 비교한다. | bcrypt 해시 검증 후 JWT 액세스 토큰 발급 | 비밀번호는 평문이 아니라 해시와 비교된다. |
| 1-6 | 로그인 성공 | 토큰과 닉네임을 저장하고 홈 화면으로 이동한다. | Zustand 상태와 `localStorage`에 `access_token`, 사용자 정보 저장 | 새로고침 후에도 로그인 상태를 유지할 수 있다. |
| 1-7 | 실패 처리 | 비밀번호 오류나 서버 연결 실패를 화면에 알려준다. | 필드 오류 또는 Toast 메시지 표시 | 사용자가 실패 원인을 이해하고 다시 시도할 수 있게 한다. |

### 2페이지. 회원가입 화면

| 세부 번호 | 설명 대상 | 화면/사용자 행동 | 내부 처리/API | 발표 포인트 |
|---|---|---|---|---|
| 2-1 | 화면 진입 | 신규 사용자가 `/register` 화면으로 이동한다. | React Router가 `Register.jsx`를 렌더링한다. | 회원가입은 서비스 사용자 계정을 만드는 단계이다. |
| 2-2 | 입력 UI | 이메일, 닉네임, 비밀번호, 비밀번호 확인을 입력한다. | 공통 입력 컴포넌트와 회원가입 폼 상태 사용 | 닉네임은 이후 댓글 작성자 표시명으로 사용된다. |
| 2-3 | 프론트 검증 | 가입 버튼을 누르면 입력 형식을 검증한다. | Zod로 이메일 형식, 닉네임 길이, 비밀번호 길이, 비밀번호 확인 일치 여부 검사 | 사용자 입력 오류를 즉시 피드백한다. |
| 2-4 | 회원가입 요청 | 검증을 통과하면 가입 API를 호출한다. | `POST /api/auth/register` | 프론트에서 검증한 데이터를 백엔드로 전달한다. |
| 2-5 | 중복 검사 | 서버가 이메일과 닉네임 중복 여부를 확인한다. | User 테이블에서 이메일/닉네임 유일성 검사 | 같은 이메일이나 닉네임으로 중복 가입을 막는다. |
| 2-6 | 비밀번호 저장 | 서버가 비밀번호를 해시로 변환해 저장한다. | passlib bcrypt, User 모델의 `hashed_password` | 비밀번호를 평문으로 저장하지 않는 기본 보안 구조를 갖춘다. |
| 2-7 | 성공/실패 흐름 | 성공 시 로그인 화면으로 이동하고 실패 시 필드 오류를 보여준다. | 성공 Toast, `/login` 이동, 400 오류 처리 | 회원가입 후 자연스럽게 로그인 흐름으로 이어진다. |

| 1 | 리포트 조회 요청 | 로그인 사용자가 상세 화면에서 리포트를 조회한다. | `GET /api/reports/{ticker}` | 리포트 열람은 보호 기능이다. |
| 2 | 기존 리포트 확인 | DB에 최신 리포트가 있는지 확인한다. | `AIReport` 테이블 | 이미 생성된 리포트는 재사용한다. |
| 3 | 생성 트리거 | 리포트가 없으면 생성 API를 호출한다. | `POST /api/ai/generate/{ticker}` | 사용자가 별도 작업을 하지 않아도 분석이 생성된다. |
| 4 | 자산 분류 | 티커를 기준으로 자산군을 분류한다. | INDEX, STOCK_US, STOCK_KR, BOND_US, BOND_KR, COMMODITY, CRYPTO | 자산군에 따라 분석 방식이 달라진다. |
| 5 | 데이터 수집 | 가격, 뉴스, 재무, 매크로 보조 정보를 모은다. | market_cache, FMP, Finnhub, CoinGecko, FRED, ECOS | 여러 출처의 데이터를 통합한다. |
| 6 | 병렬 분석 | 재무, 뉴스, 매크로 관점의 에이전트가 분석한다. | LangGraph `financial_agent`, `news_agent`, `macro_agent` | 서로 다른 관점의 분석을 모은다. |
| 7 | 자산별 분기 | 채권, 원자재, 암호화폐는 재무 분석을 생략할 수 있다. | 재무 분석 생략 조건 | 자산 특성에 맞는 분석을 수행한다. |
| 8 | 사실 통합 | 수집된 정보를 핵심 숫자, 상승 요인, 리스크 요인으로 정리한다. | synthesizer_node | 리포트 작성 전 분석 재료를 구조화한다. |
| 9 | 리포트 작성 | 한국어 Markdown 투자 리포트를 작성한다. | writer_node, OpenAI | 발표에서 보여줄 최종 본문이 생성된다. |
| 10 | 품질 평가 | 최신성, 사실성, 한국어 품질을 평가한다. | evaluator_node, 재작성 루프 | 품질이 부족하면 최대 3회 재작성한다. |
| 11 | DB 저장 | 최종 리포트를 DB에 저장한다. | `AIReport.final_content`, `bull_summary`, `bear_summary` | 이후 조회 시 빠르게 재사용한다. |
| 12 | 화면 표시 | 프론트엔드가 Markdown을 렌더링한다. | `react-markdown` | 사용자는 읽기 좋은 분석 리포트로 확인한다. |
